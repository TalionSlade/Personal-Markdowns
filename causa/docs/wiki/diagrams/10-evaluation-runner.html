EvaluationRunner: Serial vs Parallel Paths

Single agent driving N tasks; same observer surface in both paths

Causa · §I1
Serial path (parallelism = 1)
One agent, one task at a time. Easiest to inspect when a trace looks wrong. Observer events fire immediately.

Observer firing: every on_step & on_task_result fires the instant the task finishes — trace events arrive in submission order with no reordering cost.
Parallel path (parallelism > 1)
ThreadPoolExecutor fans out tasks. The runner buffers per-task trace events & emits them in submission order after each task completes — preserves human-readable JSON Lines.

Observer firing: trace events are emitted serially after the parallel run completes, in original submission order — single lock on the file handle is the only contention.

RunObserver protocol

class RunObserver(Protocol):
    def on_run_started(self, *,
                       task_count: int) -> None: ...
    def on_step(self, *, task_id: str,
                step: AgentStep) -> None: ...
    def on_task_result(self, *,
                       result: AgentResult) -> None: ...
    def on_run_finished(self) -> None: ...
